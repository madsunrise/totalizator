# Глубокая аналитика тотализатора — как воспроизвести отчёт

Здесь лежит скрипт [`analyze.py`](analyze.py), который строит большой HTML-отчёт по сезонам
тотализатора (срезы, которых нет в `/detailed_analytics`): год-к-году, мастерство vs везение,
стиль ставок, очные дуэли, суперлативы, сравнение турниров.

Скрипт **самодостаточный**: только стандартная библиотека Python 3 (никаких `pandas`/`pymongo`/
`matplotlib` — свой BSON-парсер внутри). Результат — один файл `report.html` (инлайновый CSS + SVG,
открывается офлайн в любом браузере, можно переслать в чат).

---

## Быстрый старт

```bash
# Вариант 1: данные лежат рядом со скриптом (подпапки 2024/ 2025/ 2026/)
python3 analyze.py

# Вариант 2: указать папку с данными аргументом
python3 analyze.py /Users/ivan/Downloads/totalizator_results

# Вариант 3: через переменную окружения
TOTALIZATOR_DATA=/Users/ivan/Downloads/totalizator_results python3 analyze.py
```

`report.html` пишется в ту же папку, где данные. Текущая боевая папка с данными —
`~/Downloads/totalizator_results` (там же лежит рабочая копия скрипта).

В консоль печатается **СВЕРКА** (см. ниже) и краткий дамп цифр — это нормально.

## Как выгрузить свежие данные

**BSON-дампы сезона** (на машине с MongoDB бота, `localhost:27017`; имя БД — значение
`constants.ENV_DATABASE_NAME`, коллекции `users` и `events`):

```bash
mongodump --host localhost:27017 --db "$DB_NAME" --collection events --out /tmp/dump
mongodump --host localhost:27017 --db "$DB_NAME" --collection users  --out /tmp/dump
# затем положить /tmp/dump/$DB_NAME/{events,users}.bson в <DATA_DIR>/<год>/
```

**stat.csv** — командой бота `/export_statistic` (только maintainer). Нужен лишь для кросс-проверки
2026, основной источник — BSON.

**CSV-турнир без BSON** (как Евро-2024) — достаточно выгрузки `/export_statistic` того же формата
(9 колонок: `Команда 1, Команда 2, Голы 1, Голы 2, Проход, Юзер, Ставка 1, Ставка 2, Ставка на проход`).

## ⚠️ Правила начисления очков — ГЛАВНЫЙ подводный камень

Очки считаются **каскадом** (берётся первое совпавшее, как в боте `main.py`):
`точный счёт → ничья → разница мячей → победитель`, плюс **+1 за угаданный проход** на решающих матчах.

**Веса тиров РАЗНЫЕ по сезонам** (заданы в `WEIGHTS` в начале `analyze.py`):

| Сезон | Точный | Разница | Ничья | Победитель | Проход |
|-------|:------:|:-------:|:-----:|:----------:|:------:|
| 2025  |   3    |    2    |   2   |     1      |   +1   |
| 2026  |   4    |    3    |   2   |     1      |   +1   |

Между сезонами **подорожали точный счёт (3→4) и разница (2→3)**. Поэтому «итоговые очки» сезонов
**нельзя сравнивать в лоб** — это инфляция шкалы.

> 🔴 **В НОВОМ СЕЗОНЕ ВЕСА МОГУТ СНОВА ПОМЕНЯТЬСЯ.** Не предполагай — **проверяй сверкой**.

### Как вывести/проверить веса нового сезона

Счётчики категорий (`exact / diff / draw / winner / adv_hits`) **не зависят от весов** — значит
`stored_scores = counts · weights` это линейная система. По 5+ игрокам её можно решить (least-squares),
округлить до целых и подтвердить сверкой. Именно так восстановлены веса 2025
(`exact≈3, diff≈2, draw≈2, winner≈1, adv≈1`). Практический рецепт:

1. Поставить предполагаемые веса в `WEIGHTS['<год>']`, запустить, посмотреть блок **СВЕРКА**.
2. Если расходится у всех одинаково по структуре — веса не те. Прогнать перебор целых весов
   `(exact,diff,draw,winner,adv)`, ищущий точное совпадение `stored` у всех игроков (счётчики берутся
   из `analyze_player`). Перебор занимает секунды.
3. Подобрать до совпадения, зафиксировать в `WEIGHTS`.

### Ручные правки maintainer

Если у **одного** игрока `stored` стабильно на +N выше пересчёта (как `ildariz`: +3 в 2025, +1 в 2026,
дублей ставок нет) — это **ручная корректировка очков ведущим**, а не баг. В рейтингах используется
официальный `stored`. В самом отчёте это не упоминаем (см. «Редакторские решения»).

## Гейт корректности (делать всегда)

После запуска первым делом смотри **СВЕРКУ**: пересчитанные очки (веса своего сезона) должны
совпасть со `stored` у (почти) всех. Норма — «совпало N/N, кроме ручных правок». Если совпадений мало —
не доверяй отчёту, чини веса или детект стадий. Дополнительно `stat.csv` кросс-проверяется по числу
строк-ставок.

---

## Честное сравнение сезонов

- `W_NORM` (= веса 2026) — единая шкала. Для «год-к-году» очки обоих сезонов считаются по `W_NORM`
  (поле `dnorm` в `yoy`), иначе рост правил выдаёт ложный прогресс.
- **Доли по категориям и места от весов не зависят** — это самый чистый кросс-сезонный сигнал
  мастерства (`scored_rate`, доля точных, ранги).
- Кросс-турнирно (Евро vs ЛЧ) сравниваем **места по очкам-за-ставку**, т.к. это нейтрализует и разные
  правила, и разную «лёгкость»/плотность плей-офф турниров.

## Что и где считается (карта кода `analyze.py`)

| Блок                   | Функции                                                                                                    |
|------------------------|------------------------------------------------------------------------------------------------------------|
| BSON-парсер            | `read_bson`, `_rd_doc`                                                                                     |
| Загрузка сезона (BSON) | `load_season` (понимает старое поле `is_playoff` и новое `type`)                                           |
| Очки одной ставки      | `category`, `score_bet`, near-miss: `near_miss_winner`, `near_miss_other`                                  |
| Метрики игрока         | `analyze_player` (очки, доли, проходы, near-miss/недобор, σ, серии, агрессивность, любимый счёт, энтропия) |
| Социум                 | `analyze_social` (H2H, близнецы, контрарность, сложные матчи, апсеты, победитель тура)                     |
| Клубный фаворитизм     | `analyze_club_bias`                                                                                        |
| Турнирная мета         | `analyze_tournament`, `tournament_meta_csv`                                                                |
| CSV-турнир (Евро)      | `load_csv_tournament`, `analyze_csv_player`                                                                |
| Сборка всего           | `build` → словарь `D`                                                                                      |
| Суперлативы            | `superlatives`                                                                                             |
| HTML                   | `render_html` + секции `_sec_*` (хелперы `hbar`, `stacked_cat`, `slope_chart`, `heatmap`, `table`)         |

Стадии: «решающий матч» = зафиксирован проход (`team_1_has_gone_through is not None`); лига/плей-офф —
по дате (плей-офф с февраля). Порог активности для rate-метрик — **≥30 ставок**.

## Добавить новый сезон (например, 2027)

1. Выгрузить `events.bson` / `users.bson` в `<DATA_DIR>/2027/`.
2. Добавить `WEIGHTS['2027'] = {...}` и **проверить сверкой** (см. выше). Если шкала снова сменилась
   и хочется сделать её эталоном для нормализации — обновить `W_NORM`.
3. В `build()` добавить загрузку сезона и нужные сравнения (по аналогии с 2025/2026); при желании
   расширить блоки YoY и «Сравнение турниров» на новый год.
4. Если в `events` появились новые поля стадий — дополнить `load_season`.
5. Прогнать, убедиться, что **СВЕРКА** сходится, открыть `report.html`.

## Добавить CSV-турнир (как Евро-2024)

Положить CSV в свою папку, вызвать `load_csv_tournament(path)` → `analyze_csv_player(bets, W_NORM)`,
добавить в `build()` (см. ветку `euro`) и отдельную секцию рендера (`_sec_euro`).

## Редакторские решения (важно сохранять между прогонами)

- **Джокеры**: в данных их нет (поля `is_joker` в дампах отсутствует) — в отчёте **не упоминаем**,
  это фича для будущих турниров.
- **Тон по `ildariz`**: не писать про ручные правки очков; спад подавать мягко и с контекстом
  (на Евро-2024 он был 2-м — это спад формы, не потолок).
- **Новички с малым числом ставок** (`armoald`, `forsag8_8` ≈ 10 ставок) — в общей таблице есть, но из
  rate-рейтингов («в среднем», σ, стиль) исключены порогом ≥30.
- **Евро-2024**: только CSV → очки в шкале 2026; в CSV повторные пары команд не различимы (для ЛЧ
  поэтому источник истины — BSON с `uuid`).

## «А может, я просто покажу готовый отчёт, и ты сделаешь заново?»

Можно и так — по этому README отчёт реконструируется. Но **надёжнее и быстрее**: положить свежие
выгрузки в папки по годам и перезапустить `analyze.py`. Скрипт детерминирован, сам сверяется со
`stored` и выдаёт идентичную структуру. Главное при новом сезоне — **проверить веса сверкой**.
